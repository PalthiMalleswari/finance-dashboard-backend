Overview

Backend for an organizational finance dashboard where different users interact with financial records based on their role. Supports user management, CRUD operations, aggregated analytics, and role-based access control.

Setup & installation

Follow these steps to get the project running locally.

1
Clone the repository 
git clone https://github.com/your-username/finance-dashboard-backend.git
cd finance-dashboard-backend
2
Create a virtual environment 
python -m venv venv
source venv/bin/activate          # Linux / macOS
# venv\Scripts\activate           # Windows
3
Install dependencies 
pip install -r requirements.txt
4
Create the PostgreSQL database 
psql -U postgres -c "CREATE DATABASE finance_dashboard;"
5
Configure database credentials 
export DB_NAME=finance_dashboard
export DB_USER=postgres
export DB_PASSWORD=your_password
export DB_HOST=localhost
export DB_PORT=5432
Or edit the DATABASES section in finance_project/settings.py directly.
6
Run migrations 
python manage.py makemigrations core
python manage.py migrate
7
Create a superuser 
python manage.py createsuperuser
8
Seed demo data 
python manage.py seed_data

Creates 3 users and 60 records. Credentials below.

9
Start the server 
python manage.py runserver
10
Run tests 
python manage.py test core -v 2

Filtering, search & ordering

All filters are query parameters on the /api/records/ endpoint.

ParameterDescriptionExample
entry_typeExact match?entry_type=expense
categoryExact match?category=food
date__gteFrom date (inclusive)?date__gte=2025-01-01
date__lteTo date (inclusive)?date__lte=2025-06-30
amount__gteMinimum amount?amount__gte=100
amount__lteMaximum amount?amount__lte=5000
searchText search in description & category?search=salary
orderingSort (prefix - for descending)?ordering=-amount

Combined example

GET /api/records/?entry_type=expense&category=food&amount__gte=100&ordering=-amount

Shows: food expenses over 100, sorted by highest amount first.

All errors return a consistent JSON structure via a custom exception handler.

{
    "error": true,

    "message": "Human-readable summary",
    "details": {

Design decisions & assumptions

Assumptions

AssumptionReasoning
Organizational dashboard, not personal financeRecords are shared — all users with read access see all records. The user field tracks who created the record (audit trail), not ownership. The assignment describes roles controlling actions, not data visibility.
Fixed role set (3 roles)Implemented as a TextChoices field on the User model. The assignment defines exactly three roles, so a separate Role table would add unnecessary complexity.
Predefined categoriesCategories are a fixed choice list rather than a separate table. Keeps the model simple; a dynamic system could be added later via ForeignKey.
Soft-delete for usersDELETE sets is_active = false instead of removing the row. Preserves referential integrity and allows reactivation.
No frontendBackend-only project. DRF's browsable API provides an interactive testing interface.

Key design decisions

DecisionWhy
UUID primary keysPrevents ID enumeration attacks, safe for URL exposure, works in distributed systems
Custom User model from day oneDjango recommends this before first migration — avoids painful migration later
Permission classes over decoratorsDeclarative, testable, visible at the top of each view class
Separate read/write serializersAPI inputs and outputs have different shapes (e.g., password only on create)
Database indexes on filter fieldsIndexes on entry_type, category, date, and (user, date) match query patterns
Custom exception handlerWraps all DRF errors into a uniform {error, status_code, message, details} shape

Tradeoffs

DecisionBenefitTradeoff
Session + Basic AuthSimple, works with browser and curlNot suitable for production SPAs — would need JWT
TextChoices for rolesSimple, no joinsNew roles need code change + migration
Single core appEasy to navigateWould need splitting for larger projects
Shared recordsMatches org dashboard use caseNot suitable for multi-tenant apps
SQLite-compatible queriesCan swap DB with one settingNo PostgreSQL-specific features
